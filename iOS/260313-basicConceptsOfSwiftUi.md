# [TIL] iOS 개발 환경과 핵심 프레임워크 기초

## 1. 개발 환경 세팅
### IDE (통합 개발 환경, Integrated Development Environment)
* **개념:** 소프트웨어 개발에 필요한 다양한 핵심 도구들을 하나의 GUI로 묶어 제공하는 애플리케이션이다.
* **장점:** 원래는 코드 작성, 빌드, 테스트, 디버깅 시 터미널에서 개별 도구들을 명령어로 직접 연결해야 했으나, IDE 덕분에 전체 개발 사이클을 하나의 프로그램 안에서 효율적으로 수행할 수 있게 되었다.

### Git 스테이징 (in Xcode)
* 원래 터미널에서는 `git add 파일명`을 일일이 타이핑해야 하지만, Xcode를 사용하면 Source Control 메뉴에서 변경된 파일 목록을 시각적으로 확인하고 체크박스를 클릭하는 것만으로 쉽게 스테이징 할 수 있다.
* **결론:** IDE는 복잡한 Git 명령어들을 GUI로 감싸주어 개발 편의성을 크게 높여준다.

## 2. 언어 및 보안 특징
### Swift의 멀티 패러다임
* Swift는 객체지향, 함수형, 프로토콜 지향을 모두 지원하는 **멀티 패러다임 언어**다. 
* 하나의 고정된 방식에 얽매이지 않고, 상황에 맞는 최적의 도구를 꺼내어 빠르고 안전하게 개발할 수 있다.

### 샌드박스 (Sandbox) 보안
* **개념:** iOS에서 앱이 실행될 때, 시스템이 각 앱마다 외부와 격리된 **독립적인 공간**을 할당하는 보안 원리다.
* **특징:**
  * 앱은 자신이 생성한 파일이나 설정값만 읽고 쓸 수 있다.
  * 버그나 악성 코드가 있어도 샌드박스 밖으로 나가 OS의 중요 파일을 건드리거나, 다른 앱의 데이터(채팅 내용 등)를 훔쳐보는 것이 불가능하다.
* **권한 요청 과정:** 사진, 연락처 등 샌드박스 외부의 데이터가 필요할 때는 명확한 절차를 거친다.
  1. `시스템에 요청`: 앱이 OS에게 데이터 접근 권한을 요청한다.
  2. `사용자 승인`: OS가 사용자에게 팝업을 띄워 허락을 구한다.
  3. `임시 통로 개방`: 승인 시에만 좁고 안전한 통로를 일시적으로 열어준다.
* **결론:** 샌드박스는 사용자 정보와 시스템 전체를 지켜주는 든든한 **안전 보호막**이다.

## 3. 애플의 디자인 철학: HIG
**HIG (Human Interface Guidelines)**는 앱을 처음 쓰는 사람도 헤매지 않고 편안하게 사용할 수 있도록 돕는 애플의 공식 디자인 가이드라인이다.

1. **명확성 (Clarity):** 글씨와 버튼은 한눈에 알아보기 쉽게 만들어야 한다. 텍스트는 읽기 편해야 하고, 버튼은 누르면 어떤 일이 일어날지 직관적으로 짐작할 수 있어야 한다.
2. **콘텐츠 중심 (Deference):** 진짜 주인공이 돋보여야 한다. 메뉴 바나 배경 장식 같은 UI 요소가 핵심 콘텐츠를 가리거나 시선을 빼앗아서는 안 된다.
3. **깊이감 (Depth):** 화면을 입체감 있는 3D 공간처럼 표현한다. 어떤 창이 위에 떠 있는지 시각적으로 분명하게 알려주어 사용자가 앱 내에서 길을 잃지 않게 한다.

## 4. iOS Application 프레임워크 아키텍처
iOS의 하위 계층은 복잡한 시스템 제어를 아래층에 숨기고(추상화), 개발자가 비즈니스 로직에만 전념할 수 있도록 4개의 계층으로 나뉘어 있다.

1. **Cocoa Touch (최상위):** 사용자가 직접 눈으로 보고 만지는 화면 구성(UI) 및 터치 상호작용을 담당. (UIKit, SwiftUI 포함)
2. **Media:** 시각적/청각적 생동감을 불어넣는 그래픽, 오디오, 비디오 관련 기술 계층.
3. **Core Services:** 앱 정상 작동에 필요한 핵심 데이터 처리와 논리 연산 담당. (데이터 지속성, 네트워크 통신 등)
4. **Core OS (최하위):** 기기의 하드웨어와 OS 핵심 기능을 직접 제어하는 밑바닥 계층.

### Foundation 프레임워크
Core Services 계층의 핵심으로, 어떤 앱을 만들든 무조건 필요한 **가장 밑바탕의 도구 상자**다.
* 필수 데이터 유형 및 컬렉션 (`String`, `Int`, `Array`, `Dictionary`)
* 날짜와 시간 계산 (`Date`, `Calendar`, `TimeZone`)
* 데이터 저장 및 지속성 (`FileManager`, `UserDefaults`)
* 네트워킹 (`URLSession`)
* 텍스트 처리, 정렬 및 필터링 기능

## 5. 핵심 비교: UIKit vs SwiftUI

| 구분 | UIKit (명령형) | SwiftUI (선언형) |
| :--- | :--- | :--- |
| **뷰의 정체성** | **객체 (Class):** 한 번 만들면 메모리에 계속 살아있음 | **값 (Struct):** 상태가 변하면 부수고 새로 그림 (렌더링 비용이 낮음) |
| **UI 갱신 방식** | **수동 (Manual):** 데이터 변경 시 뷰를 직접 찾아가 수정 | **자동 (Data-Driven):** 데이터와 뷰가 바인딩되어 알아서 갱신 |
| **상태 관리 도구** | Delegate, Notification, KVO 등 (복잡함) | `@State`, `@Binding`, `@Observable`(iOS 17+) 등 (직관적) |
| **결과물 확인** | 시뮬레이터 실행 또는 기기 연결 (시간 소요) | Xcode **프리뷰(Preview)**로 코드 작성 즉시 실시간 확인 |

### 💡 결론
* **SwiftUI는 데이터 주도적 설계이다:** 개발자가 수동으로 UI 업데이트 로직을 짤 필요 없이, 원하는 데이터에 `@State` 태그를 붙이고 화면(View)이 그 데이터를 바라보게(Binding) 만들면 자동으로 렌더링 된다.
* **상호 보완 관계:** 생산성과 유지보수 측면에서 SwiftUI를 우선(SwiftUI-First)으로 사용하는 것이 좋으나, 아직 모든 세밀한 제어를 완벽히 대체하지는 못하므로 UIKit 역시 반드시 알아두어야 한다.
