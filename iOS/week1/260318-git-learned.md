## 💡 학습 내용 및 깨달은 점

### 1. 폴더 인셉션(Folder Inception): 왜 같은 이름의 폴더가 겹쳐서 생성될까?
* **현상:** Tuist나 Xcode로 프로젝트를 생성할 때 `Week1_Practice` 폴더 안에 동일한 이름의 `Week1_Practice` 폴더가 또 생성되는 구조를 마주함.
* **원리 (관심사 분리):**
  * **바깥쪽 폴더 (Container):** 프로젝트의 '포장지' 역할. `.xcodeproj`, `Project.swift`(Tuist 설정도), `README.md` 등 프로젝트 전체를 아우르는 설정 파일들이 위치함.
  * **안쪽 폴더 (Source):** 프로젝트의 '알맹이' 역할. 실제 화면을 구성하는 `.swift` 코드 파일들과 리소스(이미지 등)가 담김.
* **배운 점:** 만약 포장지와 알맹이를 한 폴더에 섞어버리면, 프로젝트 규모가 커졌을 때 설정 파일과 수백 개의 코드 파일이 뒤엉켜 유지보수가 불가능해짐. 도구들이 자동으로 "설정은 밖으로, 코드는 안으로" 분리하여 소프트웨어 공학의 기본 원칙을 지켜주는 것임.

### 2. Git Root는 어떻게 결정되고 추적되는가?
* **문제 상황:** 의도하지 않은 하위 폴더가 Git의 최상단(Root)으로 설정되어, 다른 폴더들이 Git의 관리망에 들어오지 않는 문제 발생.
* **원리 (Top-Down 추적):**
  * Git은 오직 `git init` 명령어가 실행된 바로 그 위치에 `.git`이라는 숨김 폴더(본부)를 생성하며, 그곳을 우주 최상단(Root)으로 인식함.
  * Git은 본부가 설치된 폴더를 기준으로 **하위 폴더(자식, 손자) 방향으로만 내려다보며 변경 사항을 추적**함. 상위 폴더(부모)는 절대 쳐다보지 않음.
* **원인 분석:** 터미널에서 하위 폴더(예: `Week1_Practice`)에 진입해 있는 상태에서 `git init`을 실행했거나, 도구가 자동으로 초기화했기 때문에 그곳이 Root로 고정된 것임.

### 3. 다음 프로젝트를 위한 올바른 Git 연동 공식
앞으로 원하는 폴더를 정확히 Git Root로 만들고 다중 프로젝트(Workspace)를 관리하기 위한 루틴:
1. 터미널에서 기준을 삼고자 하는 **가장 최상단 폴더(Root)**로 먼저 이동한다.
2. 해당 위치에서 `git init`을 실행하여 깃 본부를 세운다. (이후 생성되는 하위 프로젝트들은 자동으로 추적망에 들어옴)
