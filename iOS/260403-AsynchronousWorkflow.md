# [TIL] 비동기 처리 흐름(async/await vs Combine)과 Race Condition

## 💡 시간이 걸리는 작업을 처리하는 두 가지 방법

앱 개발 시 네트워크 통신 등 '시간이 걸리는 작업'을 처리할 때 주로 `async/await`과 `Combine`을 사용한다. 이 둘은 일하는 패러다임 자체가 다르다.

### 1. async/await: "피자 배달 주문" 🍕
> **단일 값 반환 / 명령형(직접 제어)**

* **개념:** 피자 가게에 전화를 걸어 주문하고, 도착할 때까지 다른 일을 하며 **기다리는(await)** 방식. 피자가 도착하면 맛있게 먹고 상황이 종료된다.
* **특징:** 한 번 요청하고, 한 번 받는다. 코드가 동기 코드처럼 직관적이다.
* **언제 쓰나요?**
  * **로그인:** 아이디/비밀번호를 보내고 성공/실패 결과를 **딱 한 번** 받을 때.
  * **사진 다운로드:** 서버에서 이미지 한 장을 가져올 때.

### 2. Combine: "유튜브 채널 구독" 📺
> **연속적인 값 스트림 / 선언형(데이터 흐름)**

* **개념:** 좋아하는 채널을 '구독'해두면, 영상이 올라올 때마다 **계속해서** 알림이 오는 방식. 시간에 따라 변하는 데이터나 이벤트를 처리한다.
* **특징:** 파이프라인을 연결해 둔다. "알림 ➡️ 와이파이 확인 ➡️ 다운로드 ➡️ 화면 표시"처럼 앞으로 일어날 일들의 **규칙(체인)을 미리 세팅**해두고, 물 흐르듯 들어오는 데이터를 지속적으로 관찰한다.
* **언제 쓰나요?**
  * **실시간 검색어:** 타자를 칠 때마다 계속 변하는 입력값에 반응해야 할 때.
  * **배터리 잔량 표시:** 100% -> 99% -> 98% 등 시간에 따라 계속 변하는 값을 관찰할 때.

## 🤔 실시간 검색, 무엇으로 구현할까?

* **결론:** 최신 프레임워크 환경에서는 **Concurrency(`async/await`)** 만으로도 웬만한 처리가 가능하다.
* Combine이 데이터 흐름 제어에 강력하긴 하지만, `Swift Async Algorithms` 같은 패키지를 추가하면 Concurrency로도 Combine만큼 강력한 기능을 구현할 수 있다.

### 📌 (참고) Combine을 활용한 실습의 핵심 체인
실시간 검색을 Combine으로 구현할 때의 핵심 데이터 흐름이다. 
`@Published` 가 만든 Publisher ➡️ `debounce` ➡️ `sink` 체인으로 이어진다.

#### 🚰 쉽게 비유하자면!
* **Publisher (`@Published`):** 물이 나오는 **수도꼭지** 🚰
* **debounce:** 물줄기를 조절하는 **필터/조절 장치**
* **sink:** 물을 최종적으로 받는 **컵** 🥤

#### 1. debounce 란?
값이 너무 자주 들어올 때, 다 통과시키지 않고 **마지막 값만 남긴 채 일정 시간 기다렸다가 내보내는 기능**이다.
* **핵심:** 단순히 요청 횟수를 "줄인다"는 개념이 아니라, **"마지막 입력 타이밍을 기다린다"**는 쪽에 가깝다.
* **왜 필요할까?**
    * **API 요청 폭발 방지:** 글자를 칠 때마다 서버에 요청이 가는 것을 막아 서버 과부하를 방지한다.
    * **UX 개선:** 불필요한 검색 결과가 계속 떴다 사라지는 화면 깜빡임을 제거해 준다.

#### 2. sink 란?
Publisher에서 흘러나와 조절 장치를 통과한 값을 **"최종적으로 받아서 실제 행동을 하는 곳"**이다.
* 단순히 값을 "받는 것"으로 끝나지 않고, **사이드 이펙트를 실행하는 지점**이 된다.
    * API 호출 (서버 요청)
    * 상태 업데이트 (화면 데이터 변경)
    * 로그 출력

**💻 코드 예시:**
```swift
$searchText
    .debounce(for: .milliseconds(300), scheduler: RunLoop.main) // 필터: 0.3초 기다렸다가
    .sink { value in
        print(value) // 컵: 여기서 실제 동작(API 호출 등) 발생!
    }
```

## 🚨 Race Condition (경쟁 상태) 파헤치기

### 1. Race Condition이란?
여러 작업이 동시에 달리기 경주를 하다가, **누가 먼저 끝나느냐에 따라 결과가 뒤바뀌어 버리는 상황**을 말한다.

### 2. 검색창에서 발생하는 문제 상황
사용자가 검색창에 "app"을 빠르게 입력한다고 가정해 보자.
(`a` ➡️ `ap` ➡️ `app`)

서버로 3개의 요청이 동시에 날아간다.
* 요청 A: "a" 검색
* 요청 B: "ap" 검색
* 요청 C: "app" 검색

**🔥 문제 발생:** 컴퓨터는 입력 순서대로 처리하지 않으며 네트워크 속도도 제각각이므로 **"먼저 요청했다고 먼저 끝난다"는 보장이 없다.**
만약 속도 차이로 인해 다음과 같은 순서로 도착한다면?
* C ("app") ➡️ 빠르게 도착 ✅
* B ("ap") ➡️ 나중에 도착 ❌
* A ("a") ➡️ **제일 늦게 도착** ❌❌

결과적으로 마지막에 도착한 **A("a")의 결과가 화면을 덮어버리게 된다.** 사용자는 분명 최신 입력으로 "app"을 쳤는데 화면에는 "a"의 검색 결과가 뜨는 황당한 버그가 발생한다.

### 3. 실무에서 위험한 이유
* **검색 결과 오류:** 잘못된 데이터를 보여준다.
* **UI 깜빡임:** 결과가 뒤죽박죽 섞이며 화면이 번쩍거린다.
* **사용자 신뢰 하락:** 앱이 고장 난 것처럼 보인다.

### 4. 해결 방법
1. **이전 요청 취소 (Cancel):** 새로운 입력이 들어오면, 기존에 하던 이전 작업은 그냥 죽인다. (구조가 단순해지고 서버 낭비를 줄이며 최신 요청만 살아남는다.)
2. **최신 요청만 반영:** 응답이 도착했을 때 이게 최신 요청이 맞는지 확인하고, 아니면 버린다. (취소할 수 없는 일부 API를 사용할 때 유용하다.)

## 📝 요약 (TL;DR)
* **async/await**은 단발성 작업에, **Combine**은 지속적으로 변하는 데이터 스트림 처리에 적합하다.
* Combine의 핵심 체인은 **수도꼭지(Publisher) ➡️ 조절 장치(debounce) ➡️ 컵(sink)** 으로 이루어진다.
* **Race condition**은 순서가 뒤집히며 잘못된 결과가 나오는 문제다. 이를 해결하려면 **이전 작업을 죽이거나, 최신 것만 반영**해야 한다.
