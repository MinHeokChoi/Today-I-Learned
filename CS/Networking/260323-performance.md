## 1. 패킷 지연(Delay)은 왜 발생하는가?

라우터에 패킷이 도착하면 바로 전송되지 않고 **큐(버퍼)**에서 대기한다.  
들어오는 속도 > 나가는 속도이면 큐가 쌓이고, 큐가 가득 차면 패킷이 **버려진다(loss)**.

---

## 2. 패킷 지연의 4가지 요소

```
d_nodal = d_proc + d_queue + d_trans + d_prop
```

| 지연 종류 | 의미 | 수식 | 영향 요소 |
|-----------|------|------|-----------|
| **d_proc** (처리 지연) | 헤더 검사, 출력 링크 결정 | — | 라우터 성능 (보통 μsec 단위) |
| **d_queue** (큐잉 지연) | 큐에서 대기하는 시간 | 가변적 | 트래픽 혼잡도 |
| **d_trans** (전송 지연) | 패킷을 링크에 밀어넣는 시간 | **L / R** | 패킷 크기(L), 링크 속도(R) |
| **d_prop** (전파 지연) | 비트가 링크를 타고 이동하는 시간 | **d / s** | 링크 길이(d), 전파 속도(s) |

### 핵심: d_trans vs d_prop는 완전히 다르다!

- **d_trans**: 가래떡(패킷)을 파이프(링크) 단면에 **밀어넣는** 시간 → 패킷 크기와 링크 속도에 의존
- **d_prop**: 밀어넣어진 비트가 파이프를 **타고 이동하는** 시간 → 물리적 거리와 매체 특성에 의존

---

## 3. 단위 변환 주의사항

네트워크 문제에서 단위 실수가 가장 많으므로 반드시 체크할 것.

### 접두어 정리

| 접두어 | 읽기 | 값 | 예시 |
|--------|------|-----|------|
| **G** (대문자) | 기가 | 10⁹ | 1 Gbps = 10⁹ bps |
| **M** (대문자) | 메가 | 10⁶ | 10 Mbps = 10 × 10⁶ bps |
| **k** (소문자) | 킬로 | 10³ | 500 kbps = 500 × 10³ bps |
| **m** (소문자) | 밀리 | 10⁻³ | 20 ms = 0.02 sec |
| **μ** | 마이크로 | 10⁻⁶ | 5 μsec = 0.000005 sec |

### 자주 하는 실수

- **bytes → bits 변환**: 문제에서 L이 bytes로 주어지면 반드시 **×8** 해서 bits로!
- **km → m 변환**: d_prop 계산 시 km = 10³ m
- **m/s에서 m은 meter**(미터), **ms에서 m은 milli**(밀리) → 문맥으로 구분

## 4. End-to-End Delay 계산

N개의 링크, (N-1)개의 스위치가 있는 경로에서:

```
d_end-to-end = N × (L/R) + (N-1) × d_proc + Σ(d_i/s_i)
                전송 지연       처리 지연       전파 지연
```

### 주의: 전송 지연은 모든 링크에서 발생한다

```
A →(링크1)→ 스위치1 →(링크2)→ 스위치2 →(링크3)→ B
     d_trans      d_trans       d_trans
```

- store-and-forward 방식이므로 **링크 수만큼** d_trans 발생 (호스트 A의 첫 링크 포함!)
- 처리 지연은 **스위치 수만큼** 발생

## 5. Caravan(차량 행렬) 비유

네트워크 지연을 톨게이트와 고속도로로 비유한 모델.

| 네트워크 | 비유 |
|----------|------|
| bit | 차 1대 |
| packet (L bits) | 차 L대로 구성된 행렬 |
| 링크 전송(d_trans) | 톨부스에서 전체 행렬 처리 |
| 전파(d_prop) | 고속도로 이동 |

### 계산 예시 (슬라이드 51)

- 차 10대, 톨부스 처리 12초/대, 시속 100km, 거리 100km
- 톨부스 통과: 12 × 10 = 120초 = 2분
- 고속도로 이동: 100km / 100km/h = 60분
- **총 62분**

### Caravan 비유의 교훈 (슬라이드 52)

propagation time이 매우 짧으면 패킷의 앞부분이 다음 라우터에 도착했는데  
뒷부분은 아직 이전 라우터에서 전송 중일 수 있다.

## 6. 트래픽 강도(Traffic Intensity)

```
트래픽 강도 = La / R
```

- **L**: 패킷 크기 (bits)
- **a**: 평균 패킷 도착률 (packets/sec)
- **R**: 링크 대역폭 (bps)

| La/R 값 | 상태 | 큐잉 지연 |
|---------|------|-----------|
| ≈ 0 | 한가함 | 거의 없음 |
| → 1 | 포화 직전 | 급격히 증가 |
| > 1 | **과부하** | 큐 무한 증가 → **패킷 손실(loss)** |

### La/R > 1일 때 정확한 서술

> "도착 속도가 처리 속도를 초과하여 큐가 무한히 쌓이고,  
> 버퍼가 가득 차면 패킷 손실(loss)이 발생한다."

- ❌ "지연이 대폭 증가한다" → 불완전한 답
- ✅ 무한 큐 증가 + 패킷 손실 **두 가지를 모두** 언급해야 함

## 7. Traceroute

경로상의 각 라우터까지의 RTT(왕복 지연)를 측정하는 도구.

### 동작 원리

1. TTL(Time-to-Live)을 1, 2, 3... 으로 설정한 패킷을 전송
2. 각 라우터에서 TTL이 0이 되면 ICMP 에러 메시지를 송신자에게 반환
3. 송신자가 전송~응답 시간 간격을 측정

### Traceroute 결과 해석 포인트

- **RTT 급증**: 물리적으로 먼 구간 (예: 대서양 해저 케이블)
- **n+1번 라우터 지연 < n번 라우터 지연**: 큐잉 지연은 시간에 따라 변하므로 정상적인 현상
- **\* \* \***: 해당 라우터가 응답하지 않음 (패킷 손실 또는 응답 거부)

## 8. Throughput (처리량)

sender → receiver 간 실제 전송되는 비트의 속도 (bits/sec).

### Bottleneck Link

```
throughput = min(R₁, R₂, ..., Rₙ)
```

경로상의 링크들 중 **가장 느린 링크**가 전체 throughput을 결정한다.

### 공유 링크가 있는 경우

M개의 연결이 R bps 링크를 공유하면:

```
throughput = min(Rs, Rc, R/M)
```

### 파일 전송 시간

```
전송 시간 ≈ F / throughput
```

- F: 파일 크기 (bits) — bytes로 주어지면 ×8 변환 필수!

## 오답 노트

### 실수 1: 단위 변환 누락
- bytes를 bits로 변환(×8)하지 않음
- km를 m로 변환(×10³)하지 않음
- **대문자 M(Mega, 10⁶)과 소문자 m(milli, 10⁻³)을 혼동하지 말 것**

### 실수 2: 전송 지연(d_trans) 개수 착각
- 호스트 A에서 첫 번째 링크로 밀어넣는 것도 d_trans에 포함됨
- **링크 수 = d_trans 횟수**, 스위치 수 = d_proc 횟수

### 실수 3: 트래픽 강도 > 1 서술 부족
- "지연 증가"만 쓰면 불완전
- **큐 무한 증가 + 패킷 손실** 두 가지 모두 명시해야 함
