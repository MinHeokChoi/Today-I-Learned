# [TIL] RISC-V 함수 호출 구조와 점프 명령어의 원리

## 1. 함수 호출과 복귀의 기본: `jal`과 `ret`
* **`jal` (Jump and Link):** 함수를 호출할 때 사용하는 J-type 명령어. 현재 위치에서 지정된 라벨(오프셋)로 점프하며, 동시에 나중에 돌아올 주소(PC+4)를 `ra` (Return Address) 레지스터에 저장한다. 컴파일 타임에 목적지가 고정되어 있을 때 주로 사용한다.
* **`ret` (Return):** 호출된 함수(Callee)에서 작업을 마치고 원래 위치(Caller)로 돌아갈 때 사용한다. 이는 프로그래머의 편의를 위한 가상 명령어(Pseudo-instruction)이며, 실제 어셈블러는 이를 `jalr x0, 0(ra)`로 번역한다. 돌아갈 때는 돌아갈 주소를 다시 링크(저장)할 필요가 없으므로 목적지 레지스터로 휴지통 역할인 `x0`를 사용한다.

## 2. 동적 점프의 핵심: `jalr`
* **`jalr` (Jump and Link Register):** 레지스터(`rs1`)에 저장된 값과 12비트 오프셋을 더해 목적지를 계산하는 I-type 명령어이다. 목적지가 레지스터 값에 따라 런타임에 결정되므로, `ret`으로 돌아갈 때나 함수 포인터, `switch` 문 등을 구현할 때 필수적이다.
* 명령어의 껍데기 포맷은 `lw`와 같은 I-type이지만, 제어 장치(Control Unit)가 `opcode`를 다르게 읽고 메모리 접근이 아닌 PC(Program Counter) 변경 작업을 수행하도록 지시한다.

## 3. 먼 곳으로 점프하기: `call` 가상 명령어
* `jal` 명령어는 오프셋을 저장할 공간이 20비트로 제한되어 있어, 목적지가 너무 멀면 한 번에 점프할 수 없다.
* 이를 해결하기 위해 어셈블러는 `call offset`이라는 가상 명령어를 `auipc`와 `jalr` 두 개의 명령어로 쪼개서 처리한다.

## 4. 💡 오늘의 핵심 깨달음: 어셈블러의 부호 확장 보상 (Sign-extension Compensation)
* `call`을 쪼갤 때, 오프셋을 단순히 상위 20비트와 하위 12비트로 나누는 것이 아니라 아주 특별한 보정 작업을 거친다.
    * **문제 발생:** `jalr`이 사용하는 하위 12비트(`offset[11:0]`)의 최상위 비트(`offset[11]`)가 `1`일 경우, 컴퓨터는 이를 음수로 인식하여 부호 확장(Sign-extension)을 해버린다. 이로 인해 점프할 주소가 의도치 않게 모자라게 되는 오차(정확히 `0x1000`만큼)가 발생한다.
    * **해결책 (`auipc`의 보상):** 어셈블러는 하위 12비트에서 발생할 오차를 미리 계산하여, 1단계인 `auipc` 명령어로 상위 20비트를 계산할 때 `offset[11]` 값을 더해준다(미리 `0x1000`을 얹어줌).
    * **공식:** `auipc x1, offset[31:12] + offset[11]`

> **메타 인지 노트:** > 컴퓨터 구조에서 명령어 포맷의 한계(예: 12비트 공간의 부호 확장)가 발생시킬 수 있는 기계적인 논리 오류를, 어셈블러가 어떻게 미리 예측하고 소프트웨어적으로 보상하는지 그 치밀한 설계 원리를 이해했다!
