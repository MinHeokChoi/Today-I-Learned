content = """# TIL: 컴퓨터 구조 - 파이프라이닝(Pipelining)과 해저드(Hazards)

## 1. 파이프라이닝의 기본 원리와 성능 지표
* **Single-cycle의 한계**: 명령어 하나가 완전히 끝날 때까지 다음 명령어가 대기해야 하므로 하드웨어 자원의 낭비가 심하다. 
* **파이프라이닝(Pipelining) 개념**: 여러 명령어를 단계별로 겹쳐서(Overlap) 동시에 실행하는 마이크로아키텍처 기법이다. 세탁-건조-옷개기로 이어지는 세탁 과정에 비유할 수 있다. 
* **성능의 변화**:
  * **레이턴시(Latency, 응답 시간)**: 각 단계 사이에 위치한 파이프라인 레지스터를 거치는 오버헤드가 추가되어 개별 명령어의 실행 시간은 오히려 약간 증가할 수 있다. 
  * **처리량(Throughput, 처리량)**: 파이프라인 단계 수만큼 클럭 주기가 짧아지므로, 단위 시간당 처리하는 명령어의 수가 획기적으로 증가하여 전체 성능이 향상된다. 

## 2. RISC-V 5단계 파이프라인 구조
* **IF (Instruction Fetch)**: PC(Program Counter)를 이용해 메모리에서 명령어를 가져온다. 
* **ID (Instruction Decode)**: 명령어를 해석하고, 레지스터 파일에서 필요한 데이터를 읽어온다. 
* **EX (Execute)**: ALU를 사용하여 실제 산술/논리 연산이나 메모리 주소 계산을 수행한다. 
* **MEM (Memory Access)**: `lw`, `sw` 등의 명령어가 데이터 메모리에 접근하여 값을 읽거나 쓴다.
* **WB (Write Back)**: 최종 연산 결과나 메모리에서 읽은 데이터를 레지스터 파일에 다시 기록한다. 
* **파이프라인 레지스터 (Pipeline Registers)**: IF/ID, ID/EX, EX/MEM, MEM/WB 단계 사이에 존재하며, 이전 단계의 데이터와 제어 신호(Control Signal)를 다음 단계로 섞이지 않게 전달하는 역할을 한다. 

## 3. 파이프라인 해저드 (Pipeline Hazards)
명령어를 겹쳐 실행할 때 발생하는 문제점들을 해저드(Hazard)라고 한다. 

### ① 구조적 해저드 (Structural Hazard)
* **원인**: 여러 명령어가 동시에 동일한 하드웨어 자원(예: 메모리, 레지스터 포트 등)을 사용하려고 할 때 발생한다. 
* **해결책**: 하드웨어 자원을 추가(예: 명령어 메모리와 데이터 메모리 분리)하거나, 뒤의 명령어를 지연(Stall)시켜 해결한다. 

### ② 데이터 해저드 (Data Hazard)
* **원인**: 앞선 명령어의 실행 결과가 아직 레지스터에 기록되지 않았는데, 뒤따르는 명령어가 그 값을 읽으려 할 때 발생한다. 
* **해결책 1 (전송, Forwarding/Bypassing)**: ALU 계산이 끝난 직후(EX 또는 MEM 단계) 결과를 다음 명령어의 ALU 입력단으로 직접 넘겨주어 지연 없이 실행한다. 
* **해결책 2 (지연, Stall/Bubble)**: `lw` 명령어 직후에 그 데이터를 사용하는 명령어(Load-Use Data Hazard)가 올 경우, 데이터가 MEM 단계 이후에 준비되므로 전송만으로는 불가능하며 반드시 1클럭 지연(Stall)이 필요하다. 

### ③ 제어 해저드 (Control Hazard)
* **원인**: 분기 명령어(`beq` 등)로 인해 다음 실행할 명령어 주소가 결정되지 않은 상태에서 무작정 다음 명령어를 가져와야 할 때 발생한다. 
* **해결책 (분기 예측, Branch Prediction)**: 하드웨어가 정적 예측(항상 뜀/안 뜀) 또는 동적 예측(과거 이력 기반)을 통해 흐름을 예측한다. 
* **플러시 (Flush)**: 예측이 틀렸을 경우, 이미 파이프라인에 잘못 들어온 명령어들을 무효화(`nop` 처리)하고 올바른 주소의 명령어를 다시 가져온다.

## 4. 핵심 오개념 정리 (Reminders)
* 파이프라이닝은 명령어 하나의 속도(Latency)를 개선하는 기술이 아니라, 공장 생산 라인처럼 전체 작업의 처리량(Throughput)을 높이는 마이크로아키텍처 기술이다. 
* 제어 신호(Control Signal) 역시 데이터처럼 파이프라인 레지스터를 타고 자신이 쓰일 단계(예: WB 단계의 RegWrite)까지 전달되어야 한다.
* 분기 예측에 실패하더라도 파이프라인이 잘못된 결과를 도출하는 것이 아니라, 플러시(Flush)를 통해 약간의 사이클 페널티(시간 손해)만 지불할 뿐 정확성은 완벽히 보장된다.
