# 📝 [TIL] C 코드를 RISC-V 어셈블리로 변환할 때 자주 하는 실수 및 주의사항

오늘 C 코드를 RISC-V 어셈블리어로 변환하는 연습을 하면서, 이론으로는 알았지만 막상 코드를 짤 때 놓치기 쉬운 디테일들을 정리해 보았다.

## 1. 배열 및 메모리 주소 계산 (Memory Access)
* **상수 시프트 연산자는 `slli` 사용하기**
  * 배열의 인덱스를 바이트 오프셋(Byte offset)으로 바꿀 때 상수 2(또는 3)를 곱해야 한다. 이때 `sll`(레지스터 간 연산)이 아니라 `slli`(Immediate)를 써야 한다 [1].
* **주소 덧셈 시 엉뚱한 값 더하지 않기**
  * `slli t0, x23, 2`로 오프셋을 구했다면, 배열 시작 주소에 원본 인덱스(`x23`)가 아니라 **계산된 오프셋(`t0`)**을 더해야 한다 [2].
  * ❌ `add t0, x22, x23` (원본 인덱스를 더하는 실수)
  * ⭕ `add t0, x22, t0` (계산된 바이트 오프셋을 더함)
* **주소용 레지스터와 값용 레지스터 분리**
  * 주소를 담아둔 레지스터를 `lw`의 타겟 레지스터로 그대로 쓰면 주소 값이 덮어씌워진다. 나중에 `sw`로 다시 저장해야 할 때 주소를 잃어버리게 되므로, 주소 보관용(`t0`)과 값 계산용(`t1`)을 분리해서 쓰는 것이 효율적이다 [3, 4].

## 2. 제어 흐름 (if-else & Loops)
* **if 블록 끝에는 반드시 `else` 건너뛰기 추가**
  * C 언어와 달리 어셈블리는 위에서 아래로 그냥 흘러간다(Fall-through). `if` 블록 연산이 끝난 후 무조건 점프(`jal x0, exit`)를 해주지 않으면 `else` 블록까지 실행해버리는 치명적 버그가 발생한다 [5, 6].
* **조건문은 '반대 조건(Opposite case)'으로 점프**
  * C코드에서 `i == j`일 때 루프를 돈다면, 어셈블리에서는 `i != j`(`bne`)일 때 루프를 빠져나가도록 짜는 것이 정석이다 [7-9].

## 3. 명령어 문법 및 레지스터 최적화
* **`addi` 명령어의 올바른 문법과 `x0` 활용**
  * 문법은 `addi 도착, 시작, 상수` 순서다. 
  * 레지스터에 특정 상수를 초기화할 때는 `0`을 담고 있는 `x0` 레지스터를 활용한다 [10].
  * ❌ `addi t0, 10, 0`
  * ⭕ `addi t0, x0, 10`
* **최종 목적지 레지스터 재사용**
  * `f = g + (h - 5)`를 계산할 때, `(h - 5)`를 굳이 새로운 임시 레지스터(`t0`)에 담을 필요 없이 최종 목적지인 `f` 레지스터를 재사용하면 사용 레지스터 수를 줄일 수 있다 [11, 12].

## 4. 함수 호출과 스택 메모리 (Functions & Stack)
* **스택 공간은 사용 후 반드시 반환(Deallocation)하기**
  * Callee-saved 레지스터(`s0~s11`)를 사용하기 위해 `addi sp, sp, -4`로 스택 공간을 할당했다면, 함수 종료(`ret`) 직전에 반드시 `addi sp, sp, 4`로 스택 포인터를 원래대로 되돌려 놓아야 한다 [13-15]. 복구하지 않으면 이후 스택이 꼬이게 된다.
